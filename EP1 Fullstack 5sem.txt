2. Introducción


El presente informe tiene como objetivo diseñar una propuesta arquitectónica basada en microservicios para satisfacer las necesidades tecnológicas de la plataforma Sanos y Salvos. 
A lo largo de este documento, se describe detalladamente el contexto del problema, los requerimientos funcionales del sistema, y los patrones de arquitectura seleccionados para resolver las limitaciones actuales de la organización. Además, se justificarán las decisiones tecnológicas adoptadas y se abordarán consideraciones éticas fundamentales como la seguridad, la privacidad y la sostenibilidad del ecosistema de software.
3. Contexto y necesidades del cliente


Sanos y Salvos es una empresa dedicada a proporcionar servicios tecnológicos enfocados en la localización y recuperación de mascotas perdidas. Actualmente, la organización colabora con clínicas veterinarias, refugios y municipalidades.


En la actualidad, el registro y cruce de información se realiza de manera desorganizada y poco efectiva mayormente realizada por redes sociales. Esta fragmentación provoca el manejo de información incompleta, impide visualizar geográficamente las zonas de extravío y, de manera crítica, retrasa la detección de coincidencias entre mascotas perdidas y encontradas.


Se necesita una solución basada en microservicios que automatice la detección de coincidencias y facilite la colaboración eficiente entre los ciudadanos y las organizaciones de rescate.
4. Requerimientos funcionales


* Gestión de Mascotas: El sistema debe permitir el registro estructurado de animales perdidos y encontrados, almacenando características físicas, fotografías, ubicación geográfica y los datos de contacto del usuario.


* Sistema de Geolocalización: Debe proveer una interfaz visual (mapa) que permita a los usuarios visualizar los reportes e identificar zonas con alta incidencia de extravíos.


* Motor de Coincidencias: El sistema debe analizar automáticamente los reportes ingresados y ejecutar algoritmos para identificar coincidencias utilizando parámetros como raza, color, tamaño, ubicación y fecha.


5. Propuesta de arquitectura de microservicios


Para asegurar un sistema altamente  mantenible y capaz de escalar ante la demanda, se propone una arquitectura basada en microservicios. Esto permite que cada servicio del negocio evolucione sin afectar el funcionamiento del resto del sistema.


* Microservicio de Gestión de Mascotas: Centralizará toda la lógica de creación, lectura y actualización de los reportes estructurados de los animales.


* Microservicio de Geolocalización: Manejará el procesamiento de coordenadas espaciales y servirá la información necesaria para el mapeo interactivo en el frontend.


* Microservicio de Coincidencias (Match): Contendrá los algoritmos de búsqueda y se encargará de realizar el cruce de datos de manera asíncrona para no bloquear las operaciones de registro.


* Microservicio de Notificaciones y Alertas: Gestionará el envío de correos y alertas en tiempo real. Mediante una arquitectura orientada a eventos, reacciona automáticamente cuando el Match encuentra una coincidencia, informando a los ciudadanos e instituciones involucradas.


6. Diagrama de arquitectura 
  

El diagrama presentado expone una Arquitectura de Microservicios Orientada a Eventos, diseñada específicamente para garantizar la alta disponibilidad, escalabilidad y tolerancia a fallos que requiere la plataforma "Sanos y Salvos".
El flujo de los datos y la justificación de los componentes se estructuran de la siguiente manera:
* API Gateway Kong: Todas las peticiones de los usuarios (aplicación móvil o web) ingresan a través de Kong API Gateway. Este componente centraliza la seguridad (validación de tokens JWT) y el enrutamiento (HTTP/REST) hacia los microservicios internos, actuando como la única puerta de exposición pública de la red.
* Patrón Database per Service: Para evitar cuellos de botella y asegurar el aislamiento, cada microservicio gestiona su propia base de datos según la naturaleza de sus datos. Por ejemplo, se utiliza PostgreSQL para datos estructurados (Gestión de Mascotas), PostGIS para análisis geoespacial (Geolocalización) y MongoDB para almacenar el historial no estructurado de coincidencias (Motor de Match).
* Comunicación Asíncrona (RabbitMQ): Para evitar el antipatrón de "Monolito Distribuido", la comunicación interna principal se realiza mediante eventos asíncronos a través de un Message Broker (RabbitMQ). Cuando el Microservicio de Gestión de Mascotas registra un nuevo caso, no llama directamente a los otros servicios, sino que publica un evento ("Nueva Mascota Reportada"). El Motor de Coincidencias y el servicio de Notificaciones "escuchan" este evento y lo procesan en segundo plano. Esto asegura que el sistema siga respondiendo rápidamente al usuario sin bloqueos por tiempos de espera.
* Integraciones Externas y Tolerancia a Fallos (Resilience4j): El sistema se integra con una API externa crítica: Google Maps (para coordenadas), Dado que esta plataforma ajena puede sufrir caídas o latencias, se implementó el patrón Circuit Breaker mediante la librería Resilience4j exclusivamente en esta salida. Si la API de mapas no responde, el circuito se abre, permitiendo que Sanos y Salvos siga operando de manera local sin que los servidores colapsen esperando respuesta.
7. Patrones de arquitectura utilizados




* API Gateway: Se implementa para gestionar la comunicación entre el frontend y el backend, centralizando el acceso, mejorando la seguridad y reduciendo la complejidad en el cliente.


* Database per Service: Se utiliza para que cada microservicio gestione su modelo de datos de forma autónoma, asegurando la separación de responsabilidades.


* Repository Pattern: Se aplicará para la persistencia de datos (mediante JPA), abstrayendo la lógica de acceso a la base de datos y facilitando futuras migraciones o pruebas unitarias.


* Factory Method: Se implementará en la lógica de negocio para gestionar la creación de instancias complejas (por ejemplo, los distintos tipos de reportes y cruces de datos) estandarizando el código.


* Circuit Breaker: Resulta crítico para manejar fallos en la comunicación entre servicios. Si el Motor de Coincidencias experimenta caídas por alta demanda de procesamiento, el Circuit Breaker evitará que esta falla se propague en cascada y bloquee el Microservicio de Gestión de Mascotas.






8. Tecnologías y herramientas




* Frontend: Desarrollo de interfaces mediante React o Angular/Vue.js, empaquetando componentes como módulos NPM reutilizables y comunicación vía API REST.


* Backend: Framework Python FastAPI.


* API Gateway y Seguridad Perimetral: Se implementará Kong API Gateway como el punto de entrada principal del sistema. Esta herramienta independiente de alto rendimiento se encargará del enrutamiento de las peticiones HTTP/REST hacia los microservicios internos, además de centralizar las políticas de seguridad mediante plugins para la validación de tokens JWT y limitación de tráfico.


* Mensajería y Arquitectura Orientada a Eventos: Se utilizará RabbitMQ como Message Broker para gestionar la comunicación asíncrona entre los microservicios. Su implementación permitirá el desacoplamiento efectivo de los servicios, evitando el antipatrón de "monolito distribuido" y garantizando que las operaciones del usuario no queden bloqueadas en tiempos de espera prolongados.


* Persistencia de datos: Uso de JPA y entidades relacionales (como PostgreSQL) para garantizar el almacenamiento transaccional seguro. Posible integración de Bases de Datos NoSQL para el almacenamiento de coordenadas en el servicio de geolocalización.


* Control de Versiones: Git, implementando flujos de trabajo como Git Flow o GitHub Flow para la integración continua y el trabajo colaborativo de los desarrolladores.


9. Consideraciones éticas


* Seguridad: Las comunicaciones entre el cliente y el API Gateway estarán cifradas mediante HTTPS. La autenticación y autorización se manejará a través de tokens JWT, asegurando que solo usuarios legítimos puedan gestionar sus reportes.


* Privacidad: Debido a que se recolectarán datos de contacto y ubicaciones exactas, se implementarán mecanismos de encriptación de datos sensibles en la base de datos, asegurando la protección de los datos personales de la ciudadanía y el cumplimiento normativo.


* Sostenibilidad: A largo plazo, la arquitectura de microservicios promueve la eficiencia energética y operativa. Al ser desplegado en infraestructuras en la nube, el sistema permitirá la escalabilidad bajo demanda; es decir, solo consumirá recursos computacionales adicionales durante los peaks de reportes de extravío, reduciendo el consumo energético innecesario en periodos de bajo tráfico.


10. Conclusión


En conclusión el caso Sanos y Salvos evidencia que la implementación de una arquitectura basada en microservicios es la solución idónea para resolver sus actuales problemas de escalabilidad, fragmentación de datos y lentitud operativa. La realización de servicios desacoplados mediante un API Gateway, respaldados por patrones de resiliencia como Circuit Breaker, asegura un cumplimiento estricto de las especificaciones técnicas requeridas. Además de la adopción de prácticas de desarrollo ético garantiza la entrega de un ecosistema de software robusto, seguro, respetuoso con la privacidad de los usuarios y sostenible a nivel de infraestructura.
